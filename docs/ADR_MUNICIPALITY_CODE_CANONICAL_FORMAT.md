# ADR: Municipality Code Canonical Format（D14）

- ステータス: **Proposed**（レビュー待ち。実装・Migration・データ修正はこのADRでは一切行っていない）
- 決定日: 未定
- 起点: [PHASE4_GEOGRAPHY_MASTER_AUDIT.md](PHASE4_GEOGRAPHY_MASTER_AUDIT.md) 4節「自治体コードの保持形式」
- 目的: 全国の地理マスタ（`prefectures`/`municipalities`）を政令指定都市18市157区・将来的には
  全国約1,700市区町村へ拡張する前に、`municipalities.code`の正式な保存形式を1つに決定する

---

## 論点

`municipalities.code`に、ドキュメントが定める規約（5桁）と実際に投入されたデータ（6桁）という
**2つの異なる形式が矛盾したまま存在している**（監査4節）。全国展開でこの列に数百〜数千行を
追加する前に、どちらを正式な形式とするかを決定しないと、不整合がさらに拡大する。

---

## 1. 現状分析（5桁/6桁の不整合）

### 1-1. 事実関係（監査で確認済み）

| 項目 | 内容 |
|---|---|
| ドキュメント規約 | [全国対応データ整備ガイド.md](全国対応データ整備ガイド.md) 39行目:「`muni_code`は総務省の全国地方公共団体コード（**5桁**、例：渋谷区=`13113`）を使用してください」 |
| CSVテンプレート | `supabase/import_templates/municipalities.csv`・`public/import_templates/municipalities.csv`（同一内容）は5桁（`13113,渋谷区`／`13104,新宿区`／`13103,港区`） |
| `seed.sql`/`reset_and_seed.sql` | 渋谷区を`'13113'`（5桁）で投入 |
| `migration_organizations.sql`（Phase1.5、福岡県72件投入） | すべて6桁（例:`401013`＝北九州市門司区、`401331`＝福岡市中央区） |
| DB実データ（現時点） | 73件中72件が6桁、1件（渋谷区）のみ5桁 |

### 1-2. 6桁コードの正体を検証した（本ADR作成にあたり新たに実施）

監査時点では「6桁末尾はJIS X0402のチェックディジットではないか」という**未検証の仮説**にとどめていた。
本ADRでは、総務省が公表する全国地方公共団体コードの標準的な検査数字算出式
（5桁の各桁に重み`6,5,4,3,2`を掛けて合計し、`11`から`11で割った余り`を引く。結果が`10`なら`0`、
`11`なら`1`に読み替える）を使い、実データで再計算して検証した。

**北九州市門司区**（5桁本体`40101`と仮定）:

```
4×6 + 0×5 + 1×4 + 0×3 + 1×2 = 24+0+4+0+2 = 30
30 mod 11 = 8
11 - 8 = 3  → 検査数字 = 3
5桁本体「40101」+ 検査数字「3」 = 「401013」
```

**DBの実際の値は`401013`。完全に一致した。**

**福岡市中央区**（5桁本体`40133`と仮定）:

```
4×6 + 0×5 + 1×4 + 3×3 + 3×2 = 24+0+4+9+6 = 43
43 mod 11 = 10
11 - 10 = 1  → 検査数字 = 1
5桁本体「40133」+ 検査数字「1」 = 「401331」
```

**DBの実際の値は`401331`。完全に一致した。**

**結論（監査時点の「未確認」から更新）**: `migration_organizations.sql`が投入した6桁コードは、
**総務省の全国地方公共団体コード（JIS X0402）が定める、5桁本体＋検査数字の正規の6桁形式である**
ことを、計算によって確認した（推測ではなく検証済みの事実として記録する）。

同じ式で渋谷区の5桁本体`13113`の検査数字を計算すると:

```
1×6 + 3×5 + 1×4 + 1×3 + 3×2 = 6+15+4+3+6 = 34
34 mod 11 = 1
11 - 1 = 10  → 特殊規則により検査数字 = 0
5桁本体「13113」+ 検査数字「0」 = 「131130」
```

渋谷区の正しい6桁形式は`131130`であると計算できる（**投入するかどうかは9節「移行方針」で扱う。
本ADR自体はこの値をDBへ書き込まない**）。

### 1-3. どちらも「実在する規約」である

5桁形式（本体のみ）・6桁形式（本体＋検査数字）は、いずれも実務で使われる正当な表現であり、
「6桁が正しく5桁が誤り」という単純な話ではない。総務省が公式に「全国地方公共団体コード」として
公表するデータセットは6桁（検査数字付き）が正本だが、多くの外部システム・API・過去の資料では
検査数字を省略した5桁形式が広く使われている（このプロジェクトの`全国対応データ整備ガイド.md`も
その一例）。**問題は「どちらが正しいか」ではなく「社内で1つに統一されていないこと」である。**

## 2. 既存DBへの影響

`municipality_code`という文字列を直接保持している既存テーブルを洗い出した。

| テーブル | 保持方法 | 現在の件数 | コード変更時の影響 |
|---|---|---|---|
| `municipalities.code` | 正本 | 73件 | 変更対象そのもの |
| `anonymous_company_events` | `municipality_id`（**FKのinteger**、コード文字列ではない） | 5件（すべて`municipality_id=14`＝福岡市中央区） | **影響なし**。`id`で参照しているため、`code`列の文字列表現を変えても紐付けは壊れない |
| `workspace_companies.municipality_code` | **コード文字列そのものを保持**（`workspaceCompanyProfile.ts`で確認） | **0件**（REST確認済み、2026-07-17時点） | 現時点では影響対象が存在しない。ただしこの列は文字列一致で`municipalities`を引く（`share/[token]/page.tsx`等）ため、**将来行が増えてから`code`を変更すると、その時点のレコードは孤立する** |
| ブラウザ`localStorage`（`(site)`route groupの匿名`CompanyProfile`） | クライアント側にコード文字列を保存 | 不明（サーバー側から件数を確認できない） | Closed Beta参加者が`(site)`側を使っていた場合、保存済みの`municipalityCode`が新しい`code`値と一致せず、再入力が必要になる可能性がある。ただし[PROJECT_CONTEXT.md](../PROJECT_CONTEXT.md)により`(site)`は「互換・検証用」に位置づけが下がっており、影響範囲は限定的と考えられる |

**総括**: **今この瞬間が、コード形式を変更するコストが最も低いタイミングである。**
`workspace_companies`（正式系、Phase3以降の主経路）は0件で、`anonymous_company_events`はFK参照のため
無影響。全国展開が進み実際の顧問先データが`workspace_companies`に投入され始めると、この修正コストは
不可逆的に増加する。

## 3. Resolverへの影響

[PHASE4_GEOGRAPHY_MASTER_AUDIT.md](PHASE4_GEOGRAPHY_MASTER_AUDIT.md) 4-4節・7節の再確認: `src/lib/`配下の
`municipality_code`関連コード（`diagnosis.ts`・`events.ts`・`submissionDirectory/dataAccess.ts`等）は
すべて`.eq('code', ...)`による完全一致検索のみで、桁数を前提にした`substring`/`slice`/`padStart`等の
処理は存在しない。

**結論: どちらの形式を採用しても、Resolverコードの変更は不要。** これは「新しい`office_category`を
追加してもコード変更が不要」（ADR D13）と同じ性質であり、本ADRの決定は純粋にデータ規約の問題であって
実装の問題ではない。

## 4. CSVテンプレートへの影響

`supabase/import_templates/municipalities.csv`・`public/import_templates/municipalities.csv`は
いずれも3行のみのサンプルテンプレート（東京都3市区町村の例示）であり、全国分の完全なマスタでは
無い。どちらの形式を採用しても、**修正対象はこの3行×2ファイルのみ**であり、実装コストは小さい。

[全国対応データ整備ガイド.md](全国対応データ整備ガイド.md)は、39行目の規約説明に加え、
126〜132行目のURL確認手順の例示（`muni=13104`等）でも5桁を前提にした記述がある。6桁を採用する場合は
これらの記述も合わせて更新が必要になる。

別途作成する（本ADRでは作成しない）。

## 6. 将来の全国展開への影響

- 政令指定都市18市157区（[PHASE4_GEOGRAPHY_MASTER_AUDIT.md](PHASE4_GEOGRAPHY_MASTER_AUDIT.md) 6節）、
  将来的な全国約1,700市区町村（[NATIONAL_SUBMISSION_DIRECTORY.md](NATIONAL_SUBMISSION_DIRECTORY.md) D12、
  未着手）と、今後投入するコードの絶対量は5桁/6桁の選択に関わらず変わらない。**したがって
  「今後の投入コスト」は本ADRの決定にほぼ影響されない**（どちらの形式でも157行・1,700行を
  入力する作業量は同じ）
- 影響を受けるのは**過去に投入した73件のうち、規約と食い違っている行の修正コスト**のみ
  （2節・5節で整理した通り、6桁採用なら1件、5桁採用なら72件）
- 外部データとの突合（総務省公表の全国地方公共団体コード表、法人番号公表サイトの所在地情報等）を
  将来行う場合、総務省の一次配布形式は検査数字付き6桁が標準であるため、**6桁を正本にしておく方が
  外部データとの突合時の変換ステップが少なくて済む**（5桁を正本にした場合、外部データを都度
  5桁へ変換する処理が必要になる）

## 7. 採用候補と比較

### 選択肢A: 6桁（本体5桁＋検査数字）に統一

| 項目 | 内容 |
|---|---|
| 概要 | 既存の福岡県72件（6桁）を正本とみなし、渋谷区1件を6桁へ修正。ドキュメント・CSVテンプレートを6桁基準に更新する |
| 利点 | 修正対象が最小（1行のDB修正＋数ファイルのドキュメント修正）。総務省が一次配布する形式と一致し、将来の外部データ突合が容易。検査数字により入力ミスを機械的に検出できる（1-2節の算出式をバリデーションに使える） |
| リスク | 検査数字の算出を誤ると誤ったコードに気づきにくい（ただし1-2節の式を管理画面のバリデーションに組み込めば軽減できる、将来の実装課題として記録）。手入力時に本体5桁だけでなく検査数字も正しく付与する手間が増える |

### 選択肢B: 5桁（本体のみ）に統一

| 項目 | 内容 |
|---|---|
| 概要 | ドキュメント規約（5桁）を正本とみなし、福岡県72件を5桁へ修正する |
| 利点 | 既存ドキュメント・CSVテンプレートの修正が不要。検査数字を省くため人力での入力が単純（本体のみで済む） |
| リスク | **修正対象が72行と最大**（2節の通り、現時点ではリスクは低いが手戻り作業量は選択肢Aの72倍）。検査数字が無いため、コードの入力ミスをコード自体からは検出できない（例えば`40101`と誤って`40110`と打ち間違えても、桁数チェックだけでは気づけない）。総務省の一次データと突合する際に毎回検査数字を除去する変換が必要になる |

### 選択肢C: その他（ハイブリッド／現状追認）

| サブ案 | 概要 | 評価 |
|---|---|---|
| C1: 6桁を正本とし、5桁の「短縮表示用」列を別途追加する | `code`（6桁）+`code_short`（5桁、生成列）を持つ | 現状、5桁形式を必要とする機能（外部連携等）が存在しないため、過剰な先行投資（[CLAUDE.md](../CLAUDE.md)コーディング規約に反する）。却下 |
| C2: 現状（5桁/6桁混在）を追認し、DBレベルでの統一を諦める | 何もしない | 3節の通りResolverは完全一致検索のため直近では動作するが、監査4-4節で指摘した「新規投入者がどちらに従うべきか分からない」問題を放置することになり、全国展開が進むほど不整合が拡大する。1-1節「不整合をこれ以上拡大させない」という本ADRの目的そのものに反する。却下 |
| C3: 統一形式を決めた上で、DBに`CHECK (LENGTH(code) = 6)`等の制約を追加する | 選択肢A/Bのどちらを採ってもよいが、決定後に桁数を強制するDB制約を追加する | 将来の実装として**推奨**（9節で扱う）。ただし本ADR自体はこの制約を今回追加しない（実装禁止のため） |

## 8. 推奨案

**選択肢A（6桁、本体5桁＋検査数字に統一）を推奨する。**

### 推奨理由

1. **修正コストが最小**（2節・5節）。72件対1件という非対称性は決定的であり、選択肢Bを選ぶ理由が
   「ドキュメントを書き換えたくない」という後方互換性の都合だけになってしまう。ドキュメントの
   修正は数行の書き換えで済み、DBの72行修正よりはるかに低コスト
2. **「今が最も安いタイミング」という時間的制約**（2節）。`workspace_companies`が0件の今のうちに
   決定・修正すれば、将来の顧問先データを巻き込む心配が無い。この機会を逃すと選択肢Bのコストは
   さらに増大する
3. **総務省の一次配布形式と一致する**（6節）。将来、外部データとの突合や、より精度の高い住所解決
   機能を検討する際に変換ステップが不要になる
4. **検査数字による入力ミス検出**という副次的な利点がある（7節）。全国約1,700市区町村を人手
   （またはAIエージェント）で投入する将来のフェーズにおいて、検査数字の不一致を機械的にチェック
   できることは、[MUNICIPAL_DISCOVERY_CHECKLIST.md](MUNICIPAL_DISCOVERY_CHECKLIST.md)が重視する
   「推測による登録の禁止」「Unknown is better than Wrong」の精神とも整合する（コードの入力ミスを
   検出できないまま投入してしまうことも、一種の「誤りに気づかない」失敗であるため）

## 9. 移行方針

**本ADR自体は実装・Migration・データ修正を行わない。** 承認された場合の移行方針を以下に記録する
（実行は別タスク）。

### 9-1. 修正が必要なもの（選択肢A採用時）

| # | 対象 | 内容 |
|---|---|---|
| 1 | `municipalities`テーブル | 渋谷区の`code`を`'13113'`→`'131130'`へ更新する新規Migrationを作成（`migration_organizations.sql`等の既存ファイルは書き換えない） |
| 2 | `docs/全国対応データ整備ガイド.md` | 39行目の規約説明を「6桁（本体5桁＋検査数字）」へ、126〜132行目の例示コードを6桁へ更新 |
| 3 | `supabase/import_templates/municipalities.csv`・`public/import_templates/municipalities.csv` | 3行のサンプルを6桁へ更新（`13113`→`131130`、`13104`→`131040`、`13103`→`131030`。**これらの検査数字は1-2節と同じ式で計算するが、本ADRでは投入前提の値として提示するのみで、実際の算出・投入は別タスクで再検証すること**） |
| 4 | `docs/PHASE4_GEOGRAPHY_MASTER_AUDIT.md` 4-3節 | 「未確認の仮説」としていた記述を、1-2節の検証結果を踏まえて「確認済み」に更新する（ドキュメントの正確性維持のため。監査結果自体を書き換えるのではなく、追記・注記の形を推奨） |

### 9-2. 修正不要なもの

- `migration_organizations.sql`（福岡県72件、既存Migrationファイルは書き換えない）
- Resolver関連コード（`resolve.ts`等、3節の通り無変更）
- `anonymous_company_events`（`municipality_id`のFK参照のため無影響）

### 9-3. 今後の投入（政令指定都市18市157区・将来の全国展開）で守ること

- 新規`municipality_code`はすべて6桁（本体5桁＋検査数字）で投入する
- 検査数字は1-2節の算出式で機械的に検証してから投入する（人力での暗算・目視のみに頼らない）。
  検証手段（簡易スクリプト、または管理画面へのバリデーション追加）は本ADRでは実装しないが、
  Phase4の地理マスタMigration作成時に併せて用意することを推奨する
- 7節C3（`CHECK`制約によるDBレベルでの桁数強制）は、本ADR決定後の別Migrationとして検討する
  （本ADRでは追加しない）

### 9-4. 推奨する実施順序

1. 本ADRのレビュー・承認
2. 渋谷区1件の`code`修正Migration（9-1節#1）＋ドキュメント更新（9-1節#2〜4）を1セットとして実施
3. 修正後、`(site)`側でPlaywright確認（渋谷区の診断結果が壊れていないことを確認。[CLAUDE.md](../CLAUDE.md)の
   Playwright確認ルールに従う）
4. 確認完了後、[PHASE4_GEOGRAPHY_MASTER_AUDIT.md](PHASE4_GEOGRAPHY_MASTER_AUDIT.md) 8節の「政令指定都市
   地理マスタMigration」作成に着手する（本ADRの決定を前提にできる状態になる）

## 10. 未確定事項

1. **CSVインポート機能（`adminCsv.ts`等）が検査数字を検証する機能を持つべきか。** 本ADRでは
   「推奨する」に留め、実装要否・タイミングはプロダクトオーナー・実装者の判断に委ねる
2. **7節C3（`CHECK`制約）を本当に追加するか。** 桁数を強制する制約は、将来万一
   例外的なコード体系（例: 過去に廃止された自治体コード等）を扱う必要が生じた場合に硬直化する
   リスクがある。9-3節では「検討する」に留め、本ADRでは決定しない
3. **渋谷区の`code`修正が、Closed Beta参加者の`(site)`側`localStorage`に影響するか。** 2節の通り
   影響範囲は不明（サーバー側から確認できない）。修正実施時は、念のためClosed Beta運営側
   （[CLOSED_BETA_LAUNCH_PLAN.md](CLOSED_BETA_LAUNCH_PLAN.md)関係者）へ事前共有することを推奨する
