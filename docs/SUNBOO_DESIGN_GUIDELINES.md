## 1. ブランドの目的

SUNBOOは、経営者・税理士が

**「今年、何を・いつまでに・どこへ提出するか」**

を迷わず把握するための**行政手帳**である。会計ソフト・ERP・CRM・汎用タスク管理ではない。

提供する価値は機能数ではなく、次の5つである。

- 安心
- 明確さ
- 誠実さ
- 静けさ
- 本業へ集中できること

**ブランドメッセージ：**

> 会社を守るために、今年やるべきことを照らす。

このメッセージはUI上のあらゆる判断の基準になる。迷ったときは「これは経営者を照らしているか、それとも脅しているか／気を散らしているか」を自問する。

## 2. デザイン原則

以下を正式な原則として定義する。優先順位は番号順ではなく、すべて同格の制約として扱う。

1. **期限は探させない** — 期限情報は常に画面内で最も見つけやすい位置・大きさで表示する。
2. **一画面一目的** — 1つの画面が答える問いは1つ。複数の目的が生じたら画面を分ける。
3. **情報を増やすより、不安を減らす** — 情報量を増やす提案より、不確実性・迷いを減らす提案を優先する。
4. **分からない情報を推測しない** — データが無い・不確かな場合は「情報不足」「推定」であることを明示し、断定しない。
5. **色だけで意味を伝えない** — すべての状態表現は色＋文字／記号のペアで成立させる（色覚特性・白黒印刷でも意味が伝わること）。
6. **紙へ印刷しても読みやすい** — 画面設計の時点で「これは白黒印刷しても機能するか」を検証する。
7. **次の行動が自然に分かる** — 画面を見て「次に何をすればよいか」が説明なしに分かること。
8. **焦らせず、しかし期限は明確に伝える** — 赤・警告的な文言の多用で恐怖を煽らず、事実（日付・件数）を明確に伝えることで行動を促す。

## 3. デザインキーワード

**採用する：**

静か／温かい／誠実／整理されている／紙のような質感／朝の光／日本の手帳／控えめな上質感

**避ける：**

ERP風／会計ソフト風／派手なSaaS／Glassmorphism／Neon／Dark Mode中心／強いグラデーション／強すぎる影／大量のチャート／不必要なアニメーション

補足：Dark Modeは「中心」に据えない。OS設定に追従した最小限のダークテーマ対応（背景・文字色の反転のみ）は許容するが、専用の作り込み（Dark Mode向け新規ビジュアル要素の追加）は行わない。

## 4. カラートークン

| トークン名 | 役割 | 値 |
|---|---|---|
| Background — Warm Paper | 画面の下地 | `#FAF9F6` |
| Surface — White | カード・入力欄などの前面 | `#FFFFFF` |
| Primary — Morning Sun | 主要アクセント | `#F59E0B` |
| Text — Ink | 本文・見出し | `#1F2937` |
| Muted Text | 補助文・キャプション | `#6B7280` |
| Border — Mist | 境界線 | `#E7E5E4` |
| Success — Moss | 完了・成功 | `#6E8B74` |
| Danger | エラー・期限超過 | `#DC2626` |

### 色の運用ルール

- 白と紙色（Warm Paper／White）を主役にする。背景はWarm Paper、その上に置くカード・フォームはWhiteとし、この2階調のコントラストだけで面の重なりを表現する。
- **オレンジ（Morning Sun）は「期限」「現在地（アクティブ状態・選択中）」「主要導線（Primaryボタン）」だけに使う。** 装飾・アイコンの着色・カテゴリ分けには使わない。
- 赤（Danger）は期限超過・重大エラーだけに使う。「警告かもしれない」程度の注意喚起には使わない（→Amber系ではなくMossの反対色として弱いInk/Mutedのテキストで表現するか、既存の`caution_note`相当の中立トーンを使う。詳細は[8. コンポーネントルール](#8-コンポーネントルール)）。
- 完了は派手な緑ではなく落ち着いたMoss（`#6E8B74`）を使う。祝祭的な演出（confetti等）はしない。
- **色だけでステータスを表さず、必ず文字・記号を併用する。** アイコン単体の着色でステータスを表現しない（[16章 §9](#9-procedure-status) で現状の違反を多数確認）。
- Morning Sunは、現状 `globals.css` のBlue-600が担っている役割（リンク色・選択状態・アイコン装飾色すべて）をそのまま引き継がない。リンクや装飾アイコンは既定でInk/Mutedとし、Morning Sunは前述の3用途に絞る。

## 5. タイポグラフィ

**日本語本文：** Noto Sans JPを基本とする。

**期限の数字：** セリフ体の使用を検討してよいが、以下の制約を満たす場合に限る。
- *目的*：期限日付を「手帳に書かれた数字」のように印象づけ、他の数値（件数・パーセンテージ等）と視覚的に区別するため。
- *制約*：(1) 和文セリフ（明朝体等）ではなく、数字（0-9）とスラッシュ／ハイフンのみに限定して欧文セリフを当てる — 日本語の年月日単位（年・月・日）はNoto Sans JPのまま。(2) 本文・ラベル・ボタンには一切使わない（期限の「数字」のみ）。(3) 小さいサイズ（Caption以下）では可読性低下のため使用しない。(4) PDF・Web両方で同じフォントが利用可能であることを事前に確認する（PDF生成はNoto Sans JP埋め込み方式のため、セリフ体を採用する場合はPDF側にも別途フォント埋め込みが必要になる点に留意）。

### サイズ基準

| 役割 | サイズ |
|---|---|
| Hero | 48px |
| Page Title | 36px |
| Section Title | 28px |
| Card Title | 22px |
| Body | 16px |
| Caption | 14px |
| Tiny | 12px |

### ルール

- 日本語を英語より先に表示する。英語は補助情報として、日本語より小さく／薄く添える（例：手続き名の英語表記、会社の英語表記など）。
- 期限を画面内で最大の情報にする。少なくともHero〜Card Titleクラスのいずれかで表示し、Caption/Tinyクラスに落とさない。
- 本文の可読性を装飾より優先する。Bodyは16px以上を維持し、行間は最低1.6を確保する。
- 行間を十分に確保する（見出し級：1.3〜1.4、本文級：1.6〜1.8を目安とする）。

## 6. 余白

8pxグリッドを基準にする。

**主に使用する値：** `8 / 16 / 24 / 32 / 48 / 64`

**推奨：**

| 用途 | 値 |
|---|---|
| ページ左右余白 | 32〜64px |
| セクション間 | 48px |
| カード内 | 24〜32px |
| 情報ブロック間 | 16〜24px |

詰め込まず、余白を情報整理の一部として扱う。「もっと情報を入れたい」という要求に対しては、まず余白を削るのではなく、情報の優先順位を見直す（[原則3](#2-デザイン原則)参照）。

## 7. 角丸・境界線・影

**角丸：**

| 要素 | 値 |
|---|---|
| Card | 14px |
| Button / Input | 10〜12px |
| Badge | pill、または8px |

**境界線：** `1px solid #E7E5E4`

**影：**

| 用途 | 値 |
|---|---|
| 通常カード | `0 4px 12px rgba(15, 23, 42, 0.05)` |
| 重要カード | `0 6px 20px rgba(15, 23, 42, 0.06)` |
| モーダル | 必要最小限 |

影は装飾ではなく、**紙の重なりを表現するためだけ**に使用する。「目立たせたいから影を強くする」という判断はしない。目立たせたい場合はまず情報の優先順位（タイポグラフィスケール・配置順）で解決し、それでも不十分な場合にのみ「重要カード」の影を使う。

## 8. コンポーネントルール

### Page Header
ページ左上にPage Title（36px）または文脈に応じてSection Title（28px）。右側に主要操作を最大1つ（Primaryボタン）。パンくず・戻る導線はTinyまたはCaptionサイズ、Ink/Muted色。

### Card
White面・14px角丸・Mist境界線・通常影。内側パディング24〜32px。カード内に見出し（Card Title 22px）を置く場合、Card全体の視覚的な重さがそのままセクションの重要度を表すため、同格の情報は同格のCardスタイルで統一する。

### Roadmap Card
SUNBOOの代表コンポーネント。詳細仕様は[9. Roadmap Card](#9-roadmap-card)を参照。

### Button
原則3種類のみ。

| 種別 | 見た目 | 用途 |
|---|---|---|
| Primary | Morning Sun塗り・白文字 | 1画面に原則1つ。最も推奨する行動 |
| Secondary | White地・Mist境界線・Ink文字 | 並列する選択肢、Primaryの次点の操作 |
| Ghost | 背景・境界線なし・Ink/Muted文字 | 補助的な操作（キャンセル、閉じる、詳細を見る 等） |

主要操作は1画面1つまでを基本とする。Primaryが2つ以上必要に感じたら、画面の目的が1つに絞れていない兆候（[原則2](#2-デザイン原則)）として設計を見直す。

### Input / Select
White地・Mist境界線・角丸10〜12px。フォーカス時はMorning Sunの薄いリング（例：`box-shadow: 0 0 0 3px rgba(245, 158, 11, 0.15)`）＋境界線をMorning Sunに変更。プレースホルダー・ラベルはInk/Muted。

### Badge
カテゴリ・確からしさ（推定／情報不足）はpill形状・中立色（Ink/Muted、無彩色）。実際の状態（期限超過＝赤、完了＝Moss、要対応＝Morning Sun寄りの強調）は角丸8pxの矩形バッジとし、pill（カテゴリ）と矩形（状態）を形で区別することで「色を見なくても種類が分かる」ようにする。

### Empty State
アイコン（Mist系の淡い色）＋Body文で「まだ〇〇がありません」「条件に一致する〇〇がありません」を明示。恐怖・欠落感を与える表現（「データがありません」のみの無機質な表示）を避け、次の行動（あれば）を添える。

### Success Message
Moss系の淡い背景＋Ink文字＋チェック系アイコン。「〇〇を更新しました」のように完了した事実を簡潔に伝える。派手なアニメーションは使わない。

### Error Message
Danger（赤）は実際にユーザーの対応が必要な失敗のみに使う。「何が起きたか」「次に何をすればよいか」をセットで書く（[14. 文章ルール](#14-文章ルール)）。

### Information不足表示
データが未登録・推定である箇所は、Badge（pill、中立色）＋インライン説明を**その場に**添える。ページ冒頭でまとめて一度だけ説明する方式は、個々のバッジに立ち返ったときに意味が失われるため避ける。

### Checklist
Web上の操作チェック（クリックでステータスを更新するUI）と、PDF上の手書きチェック欄は**役割も見た目も異なるため、同一のビジュアルにしない。** Webは押せることが分かるボタン状（Secondary/Ghostに準ずる形）、PDFは実際に鉛筆で書き込める素朴な矩形の枠線のみ（詳細は[11. PDF](#11-pdf)）。

### Memo Area
罫線または淡いグレーの矩形で「余白であること」を示す。装飾を加えない。PDFでは実際に手書きできる罫線3行程度を基本とする。

### Progress
完了率などの数値は、大きなグラフではなく「今年あと◯件」のような**件数・日数の文章表現**を主とする。棒グラフ的な表現を使う場合も、Moss系の単色・細いバーに留め、パーセンテージの数字を主役にしない（[10. Dashboard](#10-dashboard)）。

### Navigation
一般ユーザー向け（Site）は上部ヘッダー＋フッターの中央寄せレイアウトを維持しつつ、背景をWarm Paperに、選択中インジケータの色をMorning Sunに変更する。管理画面（Admin）は既存の固定サイドバー構造を維持し、色のみ新トークンに置き換える。ナビゲーション自体の構造（Site/Adminの分離）は本書の対象外とし、[docs/ARCHITECTURE.md](ARCHITECTURE.md) の方針を継承する。

## 9. Roadmap Card

SUNBOOの代表コンポーネントとして、次を必須表示とする。

- 期限
- 手続き名
- 提出先
- 提出方法
- 公式ページ
- 必要書類
- ステータス
- 確からしさ

### 表示優先順位

1. 期限
2. 手続き名
3. 提出先
4. 必要書類
5. 行動・ステータス
6. 補足情報

日本語名を主表示、英語名は補助表示とする。

**必要書類は重要情報であり、可能な限り初期表示する。** 折りたたみ（アコーディオン等）の裏に隠さない。

**Web上の操作チェックと、PDF上の手書きチェックを混同しないこと。** Web版はステータス変更が可能な操作要素（Secondary/Ghost系のボタンまたはトグル）として設計し、PDF版は手で記入する枠線のみのチェック欄として設計する。両者を同一の見た目にすると、Web版のチェックが「印刷して手で書くもの」であるかのような誤解、あるいはPDF版のチェック欄が「クリックできそうに見える」誤解を招く。

### レイアウト仕様（提案）

- 1行目：期限（Card Title級、Morning Sunまたは強いInk）＋ステータスバッジ（右寄せ）
- 2行目：手続き名（Body太字、日本語主・英語補助）＋カテゴリタグ（中立pill）
- 3行目：提出先（アイコン＋Body）
- 4行目：必要書類（展開済みリスト、Caption〜Body）
- 5行目（補足）：提出方法・公式ページリンク・確からしさバッジを1行にまとめる

## 10. Dashboard

分析画面ではなく、**朝の確認画面**として定義する。

**最上部で伝えるもの：**

- 今年あと何件か
- 次の期限
- 次に行う手続き

この3点は、他のどの情報（通知・意思決定支援・進捗率など）よりも先に、1つのまとまりとして画面最上部に表示する。

**大量のチャートは禁止。** 完了率などの数値を見せる場合も、円グラフ・棒グラフの多用ではなく文章と件数で表現する（[8. コンポーネントルール — Progress](#8-コンポーネントルール)）。

**「今日やること」「近日中に行うこと」の期間定義を混同しない。** 「今日やること」は当日中に対応が必要な項目、「近日中に行うこと」はおおむね1〜2週間以内に迫っている項目とし、両者は別セクションとして明確に分ける。通知（お知らせ）・意思決定支援・進捗サマリーは、この2セクションより後ろ、かつ視覚的な優先度を下げて配置する。

## 11. PDF

PDFはレポートではなく、

**「印刷して1年間使う行政手帳」**

として設計する。

**必須：**

- 大きな期限
- 月単位の整理
- 手書きチェック欄
- メモ欄
- 提出先
- 必要書類
- 白黒印刷への対応
- ページ分断への配慮
- 会社名・所在地・対象期間
- SUNBOOブランドは控えめに表示

白黒印刷への対応は、色の濃淡だけに頼らず、文字サイズ・太字・枠線・余白で重要度を表現する設計とする。ページ分断への配慮は、1件のカードが2ページにまたがらないこと、月の区切りで改ページすることを基本とする。

## 12. Share

共有ページはLPではなく、

**「顧問先へ渡す準備済み資料」**

として設計する。

会社名とロードマップを主役にし、SUNBOOのブランドは裏方に置く。ページ最上部にSUNBOOのロゴ・ブランド名を大きく掲げず、会社名・対象期間を最初に目に入る情報とする。SUNBOOの表示は、PDFのカバーページ同様、控えめなフッター相当の扱いにとどめる。

「情報不足」「推定」の意味は、該当する項目のすぐそばで説明する。ページ冒頭に一度だけ注記して終わりにしない。

---

## 13. Profile

データベース入力画面ではなく、

**「会社の義務判定に必要な情報を登録する画面」**

として設計する。

主要項目には必ず、

**「この情報を何の判定に利用するか」**

を表示する。単なるバリデーションメッセージ（「入力してください」）ではなく、その情報がどの手続き・期限計算に使われるかを一言添える。

## 14. 文章ルール

短く、具体的で、次の行動が分かる文章にする。

**避ける：**

- 当該
- 該当する場合
- 処理に失敗しました
- エラーが発生しました
- 入力してください（単独の命令形のみで終わる場合）

**推奨：**

- 入力内容を確認してください
- 会社情報を更新しました
- 年間ロードマップを更新しました
- この情報は提出先の判定に利用します
- 10月31日までに提出しましょう

恐怖・威圧・責任転嫁を避ける。エラー文言は「何が起きたか」＋「次に何をすればよいか」をセットで書き、原因不明のまま「エラーが発生しました」で終わらせない。

**アイコン：**

Lucide系の線画。統一した線幅。原則20〜24px。

**イラスト（使用可能なモチーフ）：**

- 太陽
- 紙
- 手帳
- ペン
- 封筒
- カレンダー
- 植物

## 16. 画面監査

現行コード（`src/`）を実際に読み、以下9画面を本ガイドラインの基準で監査した。各所見には `file:line` を付す。分類はHigh/Medium/Lowの3段階（High＝「安心・明確さ」というブランド約束を損なう、またはRoadmap Cardの必須項目の可視性を妨げる／Medium＝一貫性を欠くが実害は限定的／Low＝美観上の仕上げ）。

> 前提：現行コードは全画面共通で `src/app/globals.css` の旧トークン（背景White、アクセントBlue-600 `#2563EB`、カード角丸12px・影なし、境界線 `#E5E7EB`）に依拠している。以下の各画面所見のうち「色」「角丸」「影」に関する項目の大半は、この共通トークンの置き換え（[17章 Phase 1](#17-実装計画)）で一括して解消できるため、個別画面ごとに繰り返し修正する必要はない。

### 1. Workspace Dashboard
`src/components/WorkspaceDashboard.tsx`

## 17. 実装計画

一括全面改修は禁止。既存機能・Engine・DBを変更せず、**視覚デザインだけ**を安全に置き換える前提で、以下の順に段階的実装を行う。

### Phase 1: Design Tokensと共通コンポーネント
`src/app/globals.css` の `@theme` トークン（背景・アクセント・境界線・角丸・影）を本書の値に置き換え、`.card`/`.btn-primary`/`.btn-secondary`/`.tag`/`.form-input`/`.form-select` を新トークンで更新する。Ghost用クラスの新設、Badgeのpill/矩形2系統への分離もここで行う。**この1フェーズだけで、9画面中の「色」「角丸」「影」に関するMedium所見の大半が自動的に解消される**ため、最初に着手する意義が大きい。

### Phase 2: Workspace Dashboard
情報優先順位の並べ替え（今年あと何件／次の期限／次に行う手続きを最上部に統合）、文字階層の付与、Progressの文章表現化。Phase 1のトークンに依存するため、Phase 1完了後に着手する。

### Phase 3: Workspace Roadmap（Roadmap Card）
表示順序を9章の必須順位（期限→手続き名→提出先→必要書類→ステータス→補足）に組み替え、期限をCard Title級に拡大。ステータス表現をProcedure Status統一仕様（別途要検討、[9章オープン項目](#オープン論点)参照）に合わせる。代表コンポーネントであり`(site)/roadmap`・Admin Workspace・`share/[token]`の3箇所から共有されるため、変更影響範囲が広い＝慎重に単独フェーズとする。

### Phase 4: Profile
情報構造は既に適合しているため、トークン置き換え（Phase 1で大半解消）と、チェックボックスの見た目統一のみ。実装コストが低いフェーズ。

### Phase 5: Share
ヘッダー構成の入れ替え（会社名カードを最優先表示、SUNBOOブランドをフッター相当へ縮小）。情報不足／推定の説明をバッジ単位のインライン表示に変更。

### Phase 6: PDF
構造的な作り込みは既に完成度が高いため、`COLOR_LINK`等のカラーコード差し替えが中心の軽量フェーズ。必要に応じて期限数字のセリフ体採用を検討する（[5章](#5-タイポグラフィ)の制約に従う）。

各フェーズは前フェーズの成果物（特にPhase 1のトークン）に依存するが、Engine（診断エンジン・経営イベントエンジン・ルールエンジン・通知エンジン）およびDBスキーマには一切触れない。フェーズ内で発見された「Engineの計算結果自体を変えたい」という要求が生じた場合は、本書のスコープ外として別途要件整理から起票する。

## 最終報告

### SUNBOOのデザイン原則
[2. デザイン原則](#2-デザイン原則) の8項目（期限は探させない／一画面一目的／情報を増やすより不安を減らす／分からない情報を推測しない／色だけで意味を伝えない／紙へ印刷しても読みやすい／次の行動が自然に分かる／焦らせず期限は明確に）を正式原則として定義した。

### 正式カラートークン
Warm Paper `#FAF9F6`（背景）／White `#FFFFFF`（面）／Morning Sun `#F59E0B`（アクセント、期限・現在地・主要導線限定）／Ink `#1F2937`（文字）／Muted `#6B7280`／Mist `#E7E5E4`（境界線）／Moss `#6E8B74`（成功）／Danger `#DC2626`（エラー・期限超過）。色単独でステータスを表現しないことを運用ルールとして明記。

### 正式タイポグラフィ
Noto Sans JP基本、日本語優先・英語補助。Hero48/Page Title36/Section Title28/Card Title22/Body16/Caption14/Tiny12の7段階スケール。期限数字への限定的なセリフ体採用に制約付きで言及。

8pxグリッド（8/16/24/32/48/64）。Card角丸14px、Button/Input角丸10〜12px、Badge pill/8px。境界線`#E7E5E4`。影は通常カード/重要カード/モーダルの3段階、いずれも「紙の重なり」表現に限定。

### 代表コンポーネント
Roadmap Card（期限→手続き名→提出先→必要書類→ステータス→補足の優先順位、必要書類は初期展開、Web操作チェックとPDF手書きチェックは非同一視覚）を筆頭に、Button（Primary/Secondary/Ghostの3種）、Badge（pill=カテゴリ／矩形=状態）等を[8章](#8-コンポーネントルール)で定義した。

現行コードは全画面が旧トークン（White背景・Blue-600アクセント・角丸12px・影なし）に依拠しており、新ガイドラインとは前提から異なる。個別の構造的な差としては、(1) Workspace RoadmapのCard内で期限が最も目立たない要素になっている、(2) Shareページで SUNBOOブランドが会社名より先に表示される、(3) Procedure Statusが最低5通りの異なる視覚表現で並存し統一されていない、(4) Dashboardが「朝の確認画面」ではなく通知センター先頭の分析的レイアウトになっている、の4点が最も本質的。一方PDFとProfileは構造面で既にガイドラインの意図に近く、トークン差し替えのみで対応可能。
