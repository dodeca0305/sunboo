## 1. 変更前の問題

Sprint80〜83の各監査で繰り返し指摘されていた、SUNBOOの代表コンポーネントの構造的な課題。

- 表示順が「手続き名→カテゴリ→期限→確からしさ→ステータス→提出先→必要書類」で、**期限が
  カード内で最も目立たない要素**（`text-xs text-gray-400`）になっていた
- 提出先が「提出先: 渋谷税務署」という1行のインライン表記で、ラベルと値の区別が弱かった
- 必要書類・事前準備・提出前チェックが1つの段落に読点区切りでまとめられ、見出しが分かれておらず
  スキャンしづらかった
- `caution_note`（注意事項）・`target_note`（対象）が**そもそも表示されていなかった**
  （データは取得済みだが未使用のまま）
- 読み取り専用ステータスタグに色が付いておらず、状態が一目で分からなかった（Sprint82/83で
  StatusBadge化はしたが、表示順自体は据え置きだった）
- 見出し階層が無く（手続き名が`<span>`）、スクリーンリーダーでの情報把握がしづらかった

## 2. 新しい表示構造

指示された優先順位（期限→手続き名→提出先→提出方法→公式ページ→必要書類→事前準備→
提出前チェック→Status→Confidence→補足・注意事項）に完全準拠させた。

```
┌─────────────┬───────────────────────────────────────┐
│  8月         │  源泉所得税の納付  [税務]                │
│  10          │                                         │
│  あと27日     │  提出先                                 │
│ (Desktop:左) │  🏢 渋谷税務署                           │
│              │  [公式ページ ↗]                          │
│              │  ─────────────────────────              │
│              │  必要書類 / 事前準備 / 提出前チェック       │
│              │  ─────────────────────────              │
│              │  ステータス： [StatusBadge]               │
│              │  ─────────────────────────              │
│              │  確からしさ： [StatusBadge]（該当時のみ）  │
│              │  ─────────────────────────              │
│              │  対象：／注意事項（該当時のみ）             │
└─────────────┴───────────────────────────────────────┘
```

Mobile（`<sm:`）では期限ブロックがカード上部の横並びバー（月・日・あと○日）に変わり、
以下は縦積みで変化しない。実機（375px）で確認済み（§5）。

英語名の補助表示は、`Procedure`/`ScheduleProcedure`の現行データモデルに英語名フィールドが
**存在しない**ため実装していない（Roadmapデータ構造の変更が今回禁止されているため新設もしていない）。
日本語名を主表示にする要件自体は元々満たされていた。

## 4. Status / Confidenceの共通化

- **Status**：`StatusBadge`のみを使用（独自色・独自ラベル・独自classNameは無し）。編集可能な文脈
  （Workspace管理画面）では引き続き`<select>`を使うが、`aria-label`を明示し、モバイルでは
  `min-h-11`でタップ領域を確保した。読み取り専用の文脈（公開Roadmap・Share）では
  `<StatusBadge kind={status} />`のみで表示する
- **Confidence**：`estimated`/`incomplete`のときのみ`StatusBadge`を表示し、Statusより後（優先順位10）・
  期限より視覚的に弱い位置に配置した
- **「情報不足」の説明**：`RoadmapItem.confidence`は値（`StateConfidence`）のみを持ち、
  「具体的に何のデータが不足しているか」を示すフィールドはEngine側に存在しない。**新しいフィールドを
  追加することはRoadmapデータ構造の変更にあたり今回のスコープ外**のため、既存の`/roadmap`ページの
  案内文（「プロフィールや決算情報の入力状況によって内容が変わる可能性があります」）と同じ、
  既に承認済みの一般的な説明文をカード内にも表示するに留めた。手続きごとの具体的な理由は推測せず、
  Sprint85以降でEngine側に根拠フィールドを追加する設計判断を待つ（§引き継ぎ）

- 期限ブロックがカード上部の横並びバーに変化（月・日・あと○日を横並び、下部に境界線）

- 必要書類・事前準備・提出前チェックは折りたたまず、モバイルでも初期展開のまま表示

## 6. 印刷対応

ブラウザ印刷（`window.print()`）を想定し、以下を反映した。

- カードに`break-inside-avoid`を付与し、カード途中でのページ分断を避ける
- 状態・確からしさは常にアイコン＋文言を併記しているため、背景色や罫線が印刷で消えても
  情報は残る（色のみに依存する箇所が無い）
- 必要書類は「□」（Lucide `Square`、装飾のみ・`aria-hidden`）＋テキストの組み合わせのため、
  白黒印刷でもチェック欄として機能する
- 公式ページのリンクは`<a>`のテキストラベル（例：「公式ページ」）として表示されており、URL文字列
  そのものは画面上に表示していない（既存の`buildRoadmapSubmissionInfo`の設計を踏襲）。印刷時にURLを
  読みたい場合はブラウザの印刷設定（リンクURLの併記）に委ねる形になる。これはPDF出力
  （`roadmapPdfDocument.ts`、変更禁止）が既に担っている役割であり、Web側の印刷はあくまで補助と位置づけて
  今回は追加対応しなかった

PDF生成コード（`src/lib/roadmapPdfDocument.ts`）は変更していない。

### 確認できなかったケース
以下は、開発DBの現在のデータ・プロフィール組み合わせでは該当するカードを見つけられなかった
（コードレビューでは正しく動作すると判断しているが、実データでの目視確認はできていない）。

- **期限超過**・**近日期限（14日以内）・本日締切**：`urgencyOf()`の分岐自体は単純な日数比較の
  純関数でありユニット的に正しいと判断できるが、実データでMorningSun-dark/Dangerの配色を
  画面上で確認するには至らなかった
- **必要書類・事前準備・提出前チェックがあるケース**：試した2つのプロフィール・イベント登録では
  該当データが見つからなかった。`DocumentGroup`のロジック自体はコードレビュー・型チェックで
  確認済み
- **推定・情報不足のケース**：同上。`StatusBadge kind="estimated"/"info_missing"`自体はSprint83で
  別画面（`/result`）にて実機確認済みのため、コンポーネントの表示ロジックとしては動作を確認できている
- **完了・進行中・未着手（Status）**：公開`/roadmap`は`statusMap`を渡さないためStatusセクション自体が
  非表示（既存仕様通り）。`StatusBadge`自体の3状態はSprint83で`/result`にて実機確認済み
- **管理画面（`/admin/workspaces/[id]/roadmap`）**：ログイン情報が無く実機確認不可。
  `AnnualRoadmapView`は3箇所（`(site)/roadmap`・`share/[token]`・admin workspace roadmap）から
  完全に同一のコンポーネントとして呼ばれており（分岐を作っていない）、コードレビュー上は
  `(site)/roadmap`で確認した内容がそのまま適用されるはずだが、実機での最終確認は次回に持ち越す
