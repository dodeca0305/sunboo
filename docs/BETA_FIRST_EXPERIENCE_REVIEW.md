# BETA_FIRST_EXPERIENCE_REVIEW.md — Beta First Experience Review（Sprint60）

**ステータス: UXレビューのみ。新機能追加・DB変更・migrationは一切行っていない。**

目的: Closed Beta参加者が**初回5分以内**に「何をするサービスか」「何を入力する理由か」「何が得られるか」を
理解できるかを監査する。対象はTop・診断開始・Company Profile・診断結果・Dashboard・Roadmap・Share。

---

## 0. 前提として確認した最重要事実

**実際のClosed Betaテスターは、Top・診断開始（/start）・診断結果（/result）を一度も見ない。**

`docs/CLOSED_BETA_LAUNCH_PLAN.md`（1〜4節）を確認したところ、βテスト対象者は「顧問先を持つ税理士・
会計事務所のスタッフ」であり、運営側が個別に発行したログイン情報で`/admin/login`から入る。
利用開始手順（4節）は次の通り:

```
① 管理者アカウント発行 → ② /admin/login でログイン → ③ /admin/workspaces/new で顧問先登録
→ ④ Company Profile入力（WorkspaceProfileForm） → ⑤ Roadmap確認 → ⑥ Dashboard運用
```

つまり、SUNBOOが marketing 用に用意している(site)の一連の体験（Top「経営者の参謀。」→ 診断開始
「3項目を入力するだけで...」→ 診断結果の管轄機関・手続き一覧・AI参謀）は、**実際のβ参加者の初回5分には
一切登場しない。** β参加者が最初に見る画面は`/admin/workspaces/new`という、会社名・所在地・法人種別・
決算月だけを尋ねる無地の登録フォームである（1節で詳述）。

本レビューはユーザー指定の7画面すべてを評価するが、**「診断開始」「診断結果」は(site)の`/start`・`/result`
として評価しつつ、実際のβ参加者にとっての実質的な等価画面（`/admin/workspaces/new`・Company Workspace
作成直後のDashboard）も併記する。** この前提のズレそのものが本レビュー最大の発見であるため、まず
ここに明記する。

## ② Company Profile — 入力理由の伝わりやすさ

WorkspaceProfileFormの全10項目（会社名・所在地を除く）について評価した。

| 項目 | 理由説明の有無 | 評価 |
|---|---|---|
| 法人種別 | 無し | 不十分。提出先・複数の手続き判定に直結する最重要項目の一つだが説明が無い |
| 決算月 | 無し | 不十分。全ての期限計算の起点という、Company Profile中で最も重要な項目だが説明が無い |
| 都道府県・市区町村 | **あり**（「提出先の判定に使用します」） | 十分（Sprint56で追加済み） |
| 番地・建物名 | **あり**（「判定には使用しません」まで明記） | 十分（Sprint56で追加済み） |
| 次回役員変更予定日 | **あり**（「登記期限そのものではなく...」まで明記） | 十分（Sprint55/56で追加済み） |
| 設立日 | 無し | 不十分。会社ステージ判定の起点だが説明が無い |
| 資本金 | 無し | 不十分。消費税課税事業者の特例判定（1,000万円以上）という具体的な閾値があるにもかかわらず説明が無い。**閾値そのものを画面に書けば、経営者は「なぜ資本金を聞かれるか」を1秒で理解できる** |
| 従業員数 | 無し | 不十分。社会保険・労働保険・源泉徴収関連の複数手続きを左右する重要項目だが説明が無い |
| 会社ステージ | 無し | 不十分。加えて自動推定（設立日・決算月から）と手動選択の関係が画面上で説明されていない（(site)側にはヒント文があるが、Workspace側には無い） |
| 消費税ステータス | 無し | 不十分。Rule Engineで消費税確定申告の要否を直接左右する必須項目だが説明が無い |
| インボイス登録状況 | 無し | 同上 |
| 源泉所得税の納付サイクル | 無し | 不十分。毎月納付/納期の特例で期限パターンが変わる重要項目だが説明が無い |
| 住民税特別徴収の納期 | **あり**（「年2回納付は市区町村の承認が必要」という注記のみ、amber色） | 部分的。「なぜこの情報が必要か」ではなく「誤って選ばないための注意」に留まる |
| 顧問税理士の有無 | 無し | 軽微。AI参謀の文言分岐にのみ使われる補助的な項目のため、説明が無くても実害は小さい |

**結論: 23項目中、明確な「入力理由」が書かれているのは3項目（所在地系・役員変更予定日）のみ。** それ以外の
実際にEngineの判定を左右する主要項目（法人種別・決算月・資本金・従業員数・消費税ステータス・源泉所得税の
納付サイクル）には理由説明が無い。皮肉なことに、**説明が充実しているのはSprint55・56という「最近追加された
2項目」に偏っており、既存の主要項目は最初から未整備のまま**である。

### 改善案

各項目に1行、以下のパターンで理由を添える（Sprint57監査7-1節の3パターンに準拠）。

- 決算月: 「すべての手続きの期限計算の基準になります」
- 資本金: 「1,000万円以上の場合、設立初年度から消費税の課税事業者になります」
- 従業員数: 「1人以上で源泉所得税・社会保険関連の手続きが対象になります」
- 消費税ステータス・インボイス登録状況: 「消費税確定申告が必要かどうかを判定します」
- 源泉所得税の納付サイクル: 「毎月納付か、年2回の納期の特例かで期限日が変わります」

## ③ 診断結果 — 経営者が最初に見るべき情報の順序

`/result`（`src/app/(site)/result/page.tsx`）の表示順を確認した。

1. 診断条件サマリー（入力内容の確認、青背景）
2. **管轄機関**（税務署・年金事務所等の一覧。名称・住所・電話番号・地図・公式サイトリンク）
3. **必要手続き**（`ScheduleList`: 通知カード→AI参謀カード→手続き完了率→今日/今週/今月/今後のバケット別一覧）

**経営者が最初に知りたいのは「何を・いつまでに・どこへ出すか」であり、Topページ自身が謳う価値提供の順序
（「提出書類を網羅」→「期限を自動計算」→「管轄機関を特定」→「公式リンク付き」、`(site)/page.tsx`9-30行）
は"手続き"が先、"管轄機関"は3番目である。しかし実際の`/result`画面では管轄機関が手続きより先に表示される。**
これは自社の価値提訴求の順序と、実装順序が食い違っている具体的な例である。

さらに、必要手続きセクション内部では「AI参謀」（最重要アクション）が手続き一覧より前に来ており、
これは適切（一番知りたい「今何をすべきか」が最初に来ている）。問題は「管轄機関ブロックがAI参謀より
さらに手前にある」ことにある。

### 改善案

「必要手続き」（AI参謀の最重要アクションを含む）を先に、「管轄機関」を後にする表示順の入れ替えを推奨する
（コード変更を伴うためレビュー後に実施）。管轄機関は「どこに出すか」という参照情報であり、各手続きの
詳細展開（`ProcedureRow`）内でも個別に提出先の情報（`proc.office`）を表示できる設計のため、トップの
専用セクションとしては後段に回しても情報は失われない。

## ④ Roadmap — 「何を・いつ・どこへ」の理解しやすさ

`AnnualRoadmapView.tsx`のカード1件（1手続き×1回分の出現）を確認した。

- **何を**: 手続き名・カテゴリタグが同じ行に表示される（○）
- **いつ**: 期限日付が同じ行に表示される（○）。ただしConfidenceが低い場合の「推定」「情報不足」タグも
  同じ行にあるため、初見で「期限が確定していない可能性がある」ことに気づけるかは注意深さに依存する
- **どこへ**: 提出先名（`submission.officeName`）は表示されるが、**手続き名・期限と同じ行ではなく2行目**
  （172-177行）にあり、視線を1段下げないと分からない。提出先が未登録の場合は「提出先情報は未登録です」
  という別文言になる

必要書類（`hasDocGuide`のブロック）はカードの一番下、3行目以降に折りたたまれず常時表示される
（13手続きにしか登録が無いため多くのカードには出ない、Sprint57/`BETA_BACKLOG.md` L-05既知）。

**結論: 1件のカードだけを見れば「何を・いつ・どこへ」は揃っているが、「どこへ」が視覚的に一段下がって
いるため、一覧をざっと流し読みするとき「いつ」までしか目に入らず、「どこへ」を見落とすリスクがある。**
また、ページ全体としては「これは何のページか」を説明する導入文が無く、初見のβ参加者はカードの並びから
自力で「年→月→手続き」という構造を読み取る必要がある。

### 改善案

- ページ冒頭に1行、「今後2年分の手続き予定を、期限順に一覧表示しています。手続き名の下に提出先・
  必要書類が続きます」という導入文を追加する（既存の免責文とは別に、機能説明の文を先に置く）
- 提出先を手続き名・期限と同じ行（またはすぐ隣接する行）に寄せるレイアウト調整を検討する

## ⑤ Dashboard — 情報量の適切さ

`WorkspaceDashboard.tsx`は「通知センター」「今日やること」「期限警告」「意思決定」「進捗サマリー」
「AI参謀」「会社概要」の7区画を持つ。

- **不要と判定したカード: 無し。** 各区画は役割が明確に分離されている（コードコメントに設計意図が
  明記されており、実際に重複した情報は無い。Sprint59監査で確認済み）
- **情報量として過多に見えるリスク**: 初回（Company Profile未入力）時点では7区画のうち実質的に
  意味のある内容を持つのは「会社概要」のみで、残り6区画は「対応が必要な通知はありません」
  「直近で対応が必要な手続きはありません」等の空メッセージが並ぶ。**初回訪問時に7つの空カードが
  縦に並ぶ状態は、情報量として少ないのではなく「何も無いことを7回言われる」という別の意味で
  圧迫感がある**
- **不足**: Company Profileが未入力であること自体への直接的な誘導が無い。「意思決定」区画は
  `state.stage.confidence==='incomplete'`のケースを扱うが、これは`generateWorkspaceDecisions`内部の
  ロジックであり、初回訪問者にとって最も欲しい「まずプロフィールを入力してください」という
  最優先の1文がどの区画にも無い

### 改善案

初回（Roadmap計算結果が0件、または`state.stage.confidence==='incomplete'`）の場合に限り、7区画の
代わりに「Company Profileを入力すると、ここに通知・today's action・年間予定が表示されます」という
単一の誘導カードを表示する分岐を追加する案を推奨する（コード変更を伴うためレビュー後に実施）。

## ⑥ Topページ — 30秒で価値が伝わるか

`src/app/(site)/page.tsx`を確認した。

- **Hero**: 「経営者の参謀。」（キャッチコピー、抽象的）→「会社ごとに必要な手続き・提出期限・提出先を
  整理し、経営者が本業に集中できる環境をつくります。」（説明文、具体的）→ CTA（はじめる/デモとして
  試す/手続きを検索）
- **3ステップ**: 会社情報を入力（3項目）→ 診断を実行 → 期限と提出先を確認
- **4つの特徴**: 提出書類を網羅／期限を自動計算／管轄機関を特定／公式リンク付き

**評価: 30秒以内に価値は伝わる構成になっている。** キャッチコピー単独では抽象的だが、直後の説明文・
3ステップ・4特徴が畳み掛けるように具体化しており、上から順に読めば「手続き・期限・提出先の整理サービス」
という理解に30秒以内で到達できる。「デモとして試す」ボタンでフォーム入力無しに`/result`へ直接遷移できる
導線も、価値を体感させる工夫として機能している。

**課題**: Topページはβ参加者（税理士・会計事務所スタッフ）が実際には見ないページである（0節）。
一般公開後の集客用ページとしては妥当だが、**β参加者向けの価値説明としては機能していない**。⑧の
3分ガイドで補う必要がある。

### 改善案

- キャッチコピー「経営者の参謀。」を、税理士・会計事務所も対象読者に含めるなら「顧問先の参謀。」等の
  併記、もしくはβ期間中はβ参加者向けランディング文言を別途用意することを検討する
- Topページ自体の大きな改善は不要（③④⑤と異なり、構成として弱点は無い）

## ⑦ Share — 受け取った経営者が説明なしで理解できるか

`/share/[token]`（`src/app/share/[token]/page.tsx`）を確認した。

- ヘッダー: SUNBOOロゴ＋「共有ページ（閲覧専用）」タグ → 会社名・法人種別タグ・所在地・決算月
- 本文: 「年間ロードマップ」見出し＋説明文（「今年度から今後2年分の手続き予定を一覧表示する参考情報
  です。実際の手続き・期限・提出先は必ず顧問の専門家・各公式機関の最新情報をご確認ください。」）
  → `AnnualRoadmapView`（Roadmapと同じカード表示）
- フッター: 免責文（記帳・電子申告・法的助言ではない旨）

**評価: 「これが何のページか」は説明文で伝わる（年間ロードマップ＝手続き予定の一覧、という説明が
本文中にある）。** ただし以下の点で「説明なしで完全に理解できる」とは言い切れない:

- 送付元（誰が・なぜこのリンクを送ってきたか）の説明が無い。税理士から届いた場合、経営者は
  「これはSUNBOOという何のサービスなのか」を文脈から推測する必要がある
- Roadmapと同じ課題（④）を継承しており、「どこへ」提出するかがカード内で視覚的に一段下がっている
- Confidence（推定/情報不足）タグの意味が、Share画面内では一切説明されていない（Workspace側の
  Roadmapページには「実際の手続き・期限・提出先は...「情報不足」「推定」の表示がある手続きは...」
  という注記があるが、Shareページの説明文にはConfidenceタグへの言及が無い、`share/[token]/page.tsx`
  119-122行）

### 改善案

Shareページの説明文に、送付元を示す1文（例:「◯◯様よりSUNBOO経営ナビの年間ロードマップが共有されました」
のようなテンプレート、company名を使った動的文言）と、Confidenceタグの意味（「推定」「情報不足」の
簡単な説明）を追加することを推奨する（コード変更を伴うためレビュー後に実施）。

## まとめ

### 最優先改善3件

1. **Company Profileの主要項目（法人種別・決算月・資本金・従業員数・消費税ステータス・源泉所得税の
   納付サイクル）に入力理由の説明を追加する**（②）。現状、実際にEngineの判定を左右する項目のほとんどに
   理由説明が無く、Sprint55/56で追加した3項目だけが手厚いという偏りがある
2. **β参加者向けの3分ガイドを作成・配布する**（⑧）。現状、β参加者はTopページの価値説明を一度も見ずに
   無地の登録フォームへ直接入るため、サービスの全体像を伝える手段が実質的に存在しない
3. **`/result`の表示順（管轄機関→必要手続き）を、Topページが謳う価値提供順（手続き→期限→管轄機関）に
   合わせて入れ替える**（③）。経営者が最初に見るべき「何を・いつまでに」が、参照情報である管轄機関一覧の
   後ろに追いやられている

### 改善不要3件

1. **Topページの構成**（⑥）。キャッチコピー→説明文→3ステップ→4特徴という流れは30秒以内に価値が伝わる
   構成として機能しており、大きな変更は不要
2. **Dashboardのカード構成**（⑤）。7区画はいずれも役割が明確で重複が無く、「不要なカード」は無い。
   改善が必要なのは情報量ではなく初回（空状態）の見せ方
3. **Excel/PDF/Roadmapの「何を・いつ」の伝わりやすさ**（④）。手続き名・カテゴリ・期限は同じ行に
   揃っており、この2点については初見でも迷わない（「どこへ」の視認性のみ改善余地あり）

### β開始可否

**Goだが、開始前に最優先改善3件のうち②（3分ガイドの配布）だけは実施を強く推奨する。** ①・③はコード
変更を伴うため次Sprintでの対応で構わないが、③3分ガイドはドキュメント作成のみで完結し、かつ0節で
明らかになった「β参加者はSUNBOOの価値説明を一度も見ないまま無地のフォームに入る」という構造的な
ギャップを埋める最も費用対効果の高い対応である。Blocker相当（機能が壊れている・誤情報を与える）の
問題は本レビューでは発見していない。
